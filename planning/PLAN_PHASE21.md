# Phase 21 — Research Rework

## Overview

This phase replaces the cumulative field-point research model with a per-turn spending model, introduces a technology dependency graph, adds cross-field breakthrough mechanics, and reworks the right-column UI to focus purely on Signal and Research. The Board panel moves to the main map area.

This is the largest single rework since Phase 14. It is broken into sub-phases that can be implemented and tested independently, though sub-phases 21.1–21.4 are tightly coupled and should be completed together before any UI work begins.

---

## Sub-phases

### 21.1 — Data model: per-tech field progress

**Files:** `src/engine/types.ts`, `src/engine/state.ts`, `src/engine/save.ts`

**The core model shift:** Field points no longer accumulate globally in `player.fields`. Instead, each `TechState` tracks how many field points it has received so far toward its recipe. The global `player.fields` is repurposed to hold this turn's field output (a per-turn snapshot for display and breakthrough checking), not a running total.

#### Changes to `types.ts`

Add `fieldProgress` to `TechState`:

```typescript
export interface TechState {
  defId: string;
  stage: TechStage;           // 'unknown' | 'rumour' | 'progress' | 'discovered'
  recipe: TechRecipe;         // field thresholds (already exists, seed-randomised)
  fieldProgress: TechRecipe;  // accumulated points per field toward this tech's recipe
  unlockedByBreakthrough: boolean; // true if Rumour was triggered by a breakthrough condition
}
```

`player.fields` keeps the same type (`FieldPoints`) but its semantics change: it now holds the total field output generated in the most recently completed World Phase (for display in the HUD and for breakthrough condition evaluation). It is overwritten each turn, not accumulated.

#### Changes to `state.ts`

- `createTechState` initialises `fieldProgress` as all-zeroes
- `createGameState` / `createPlayerState` unchanged structurally

#### Changes to `save.ts`

- `serialiseGameState` / `deserialiseGameState`: include `fieldProgress` and `unlockedByBreakthrough` in `TechState`
- `validateSave`: extend tech state validation to check these new fields
- **Save migration**: existing saves will have `TechState` entries without `fieldProgress`. `deserialiseGameState` should default missing `fieldProgress` to `{}` rather than failing.

### 21.2 — Engine: research distribution algorithm

**Files:** `src/engine/research.ts`, `src/engine/turn.ts`

This is the heart of the new model. Each World Phase, the field output generated by facilities is distributed across all applicable technologies.

#### New function: `distributeResearchPoints`

```typescript
export function distributeResearchPoints(
  techs: TechState[],
  techDefs: Map<string, TechDef>,
  fieldOutput: FieldPoints,  // this turn's total facility output
  rng: Rng,
): TechState[]
```

**Algorithm:**

1. **Identify applicable techs**: techs at `'rumour'` or `'progress'` stage that still have at least one field in their recipe below threshold.

2. **For each applicable tech, for each field in its recipe**: guarantee a minimum of 1 point (or `tier × eraMultiplier` — see open questions) if that field has any output and the tech still needs it. Apply these minimums first, deducting from `fieldOutput`.

3. **Distribute remaining points randomly**: for each field, if output remains after guarantees, pick a random applicable tech that still needs that field and assign the remainder. Repeat until field output is exhausted or no applicable tech needs it.

4. **Points that cannot be spent** (all applicable techs have met a field's threshold) are simply discarded for that turn.

5. **Return updated `TechState[]`** with new `fieldProgress` values.

#### Changes to `turn.ts` — `executeWorldPhase`

Replace the current `applyFieldDeltas` accumulation with a call to `distributeResearchPoints`. The computed `fieldOutput` (from `computeFacilityOutput`) is passed both to `distributeResearchPoints` and stored in `player.fields` for display. The PRNG call for research distribution must be added to the call-order comment block.

#### Remove from `research.ts`

- The current `checkResearchProgress` function's field-accumulation logic. Discovery stage checking is now a separate concern (see 21.3) and no longer reads from `player.fields`.

### 21.3 — Engine: discovery stage transitions

**Files:** `src/engine/research.ts`, `src/engine/turn.ts`

With per-tech `fieldProgress` now accumulating, stage transitions are evaluated after `distributeResearchPoints` runs each turn.

#### Updated `getDiscoveryStage`

```typescript
export function getDiscoveryStage(
  tech: TechState,
  def: TechDef,
  allTechs: TechState[],         // needed to check prerequisite stages
  techDefs: Map<string, TechDef>,
): TechStage
```

**Transition rules:**

| To stage | Condition |
|---|---|
| `'rumour'` | All prerequisite techs are at `'progress'` or `'discovered'`; OR tech is Tier 1 (no prerequisites); OR breakthrough triggered (see 21.6) |
| `'progress'` | Tech is at `'rumour'` AND: all required fields ≥ 33% of recipe threshold, OR at least 33% of required fields ≥ 50% of threshold |
| `'discovered'` | All required fields in `fieldProgress` ≥ recipe threshold |

The `'unknown' → 'rumour'` transition is evaluated separately in the prerequisite check (21.4). The `'rumour' → 'progress'` and `'progress' → 'discovered'` transitions are checked each turn after distribution.

**Note on Tier 1 at game start:** All Era 1 Tier 1 techs begin at `'rumour'` (set in `startNewGame` rather than by this function). This is a special initialisation, not an ongoing rule.

#### Changes to `turn.ts`

After `distributeResearchPoints`, iterate all techs and apply stage transitions. Stage changes from this evaluation are used to trigger news items and narrative queues (as currently happens in `advancePhase` in the store).

### 21.4 — Engine: tech prerequisites

**Files:** `src/engine/types.ts`, `src/engine/research.ts`, `src/data/technologies.ts`

#### Changes to `types.ts`

Add `requiredTechIds` to `TechDef`:

```typescript
export interface TechDef {
  // ... existing fields ...
  requiredTechIds: string[];  // IDs of techs that must be at 'progress' or 'discovered' to unlock this one as a Rumour. Empty array = no prerequisites (Tier 1 or starting techs).
}
```

#### Changes to `research.ts`

New helper:

```typescript
export function prerequisitesMet(
  techDef: TechDef,
  allTechs: TechState[],
): boolean
```

Returns `true` if every entry in `requiredTechIds` is at `'progress'` or `'discovered'`. Used by `getDiscoveryStage` for the `unknown → rumour` check.

#### Changes to `technologies.ts`

All 12 existing `TechDef` entries get `requiredTechIds: []` initially (Tier 1) or appropriate prerequisite arrays (Tier 2+). The full prerequisite graph is defined in sub-phase 21.5.

2. Prerequisite chains (`requiredTechIds`)
3. `requiredTechIds: []` added to satisfy the new type

### 21.6 — Engine: breakthrough tech mechanics

**Files:** `src/engine/types.ts`, `src/engine/research.ts`, `src/engine/turn.ts`

#### Changes to `types.ts`

```typescript
export interface BreakthroughCondition {
  // All of these must be true simultaneously on the same turn:
  fieldOutputThresholds: Partial<FieldPoints>; // e.g. { physics: 20, computing: 15 }
  facilityDefIds?: string[];                    // e.g. ['computingHub'] — at least one of each must be active
  facilityCount?: number;                       // minimum total active facilities (optional)
}

export interface TechDef {
  // ... existing fields ...
  breakthroughCondition?: BreakthroughCondition; // only on Tier 3+ techs
}
```

`TechState.unlockedByBreakthrough` (added in 21.1) records whether this tech entered `'rumour'` via its breakthrough condition rather than via prerequisites.

#### New function: `checkBreakthroughConditions`

```typescript
export function checkBreakthroughConditions(
  techs: TechState[],
  techDefs: Map<string, TechDef>,
  fieldOutput: FieldPoints,         // this turn's output (from player.fields)
  activeFacilityDefIds: string[],   // defIds of all non-queued, non-demolishing facilities
): BreakthroughResult[]
```

Returns an array of `{ techId, wasAlreadyRumoured }` for each tech whose breakthrough conditions are newly met this turn. Only triggers if the tech is currently `'unknown'`.

#### Changes to `turn.ts`

Call `checkBreakthroughConditions` during `executeWorldPhase`, after facility output is computed and before `distributeResearchPoints`. For each triggered breakthrough:
- Promote the tech to `'rumour'`, setting `unlockedByBreakthrough: true`
- Enqueue a news item: e.g. _"Breakthrough: unexpected convergence of physics and computing research has revealed a new avenue of investigation."_

**Which techs get breakthrough conditions:** Only 2–3 techs across Era 1 (Tier 3 or 4 only). Candidates: `signalPatternAnalysis`, `orbitalMechanics`, possibly one Tier 3 tech. The normal prerequisite path must always remain open — breakthrough is an _earlier_ route, never the _only_ route.

### 21.8 — UI: in-progress tech summary and Tech Tree button

**Files:** `src/lib/components/ResearchFeed.svelte` (heavily reworked), new `src/lib/components/TechProgressSummary.svelte`

#### TECH TREE button

The button currently lives in the ResearchFeed header at small size. In the reworked right column it becomes a more prominent standalone element — full-width button, clearly labelled. It gains a visual indicator (a pulsing dot or badge count) when one or more techs transitioned to `'rumour'` or `'progress'` this turn. The indicator clears when the tech tree is opened.

```typescript
// New reactive state in game store (or derived in +page.svelte):
hasNewRumours: boolean  // true if any tech became 'rumour' since tech tree was last opened
```

#### New component: `TechProgressSummary.svelte`

Shows the up-to-3 techs currently at `'progress'` stage, ordered by overall completion (closest to discovered first). For each tech:

- Tech name
- One thin progress bar per required field, colour-coded by field (see 21.9), showing `fieldProgress[field] / recipe[field]` as a percentage
- No numeric values — the bars are visual indicators only
- Small field-name label (abbreviated: PHY, MTH, ENG, etc.) beside each bar

If fewer than 3 techs are at `'progress'`, the remaining slots are either empty or show "No further research in progress."

This component receives `techs: TechState[]`, `techDefs: Map<string, TechDef>` as props.

#### Removal from ResearchFeed

Remove:
- The six field accumulation bars
- The field numeric values
- The existing TECH TREE button (replaced by the standalone button above)

The wormhole climax section stays in ResearchFeed (or is moved to a dedicated component if it grows — deferred decision).

### 21.10 — Tech tree: dependency arrows

**File:** `src/phaser/TechTreeScene.ts`

**Prerequisite:** 21.4 must be complete (so `TechDef.requiredTechIds` is populated).

Draw curved spline arrows between tech nodes to visualise the dependency graph. This is a pure rendering change to the existing Phaser scene — no engine or store changes needed.

#### Rendering approach

After all nodes are positioned, for each tech with `requiredTechIds.length > 0`, draw a bezier curve from the right edge of each prerequisite node to the left edge of the dependent node.

```typescript
// For each dependency edge (sourceNodeCenter → targetNodeCenter):
const cp1x = source.x + (target.x - source.x) * 0.5;
const cp1y = source.y;
const cp2x = source.x + (target.x - source.x) * 0.5;
const cp2y = target.y;

graphics.strokePath();  // draw bezier through control points
```

**Visual treatment:**
- Lines are drawn on `worldGfx` (scrolls with camera) beneath node rectangles
- Colour: dim (`#1e3040`) for undiscovered/unknown prerequisites; brighter (`#2a6090`) when the prerequisite is `'discovered'`; bright white-blue (`#6aacca`) when the connection is "active" (prerequisite discovered, dependent in progress)
- Arrow tip: small filled triangle at the target end
- Lines route behind overlapping nodes — draw all edges before drawing nodes

#### Breakthrough indicator

If `tech.unlockedByBreakthrough === true`, the node gets a subtle visual marker (a small lightning-bolt glyph or a distinct corner indicator) indicating an alternative unlock path was used. This is only shown if the breakthrough *did* fire — techs unlocked via the normal path show no indicator, preserving the surprise for future playthroughs.

## Outstanding questions and concerns

**1. Minimum guaranteed points scaling**
The draft specifies "at least one point per applicable tech per field." With 8+ techs in progress simultaneously in late Era 1, the minimum guarantee alone consumes a large fraction of total field output. Playtesting will be needed to determine whether the minimum should scale with tier/era, or whether it should remain flat at 1. Define this as a named constant so it is easy to tune.

- _Agreed. Define this as a named constant_

**2. What `player.fields` means after the change**
Several existing engine and UI paths read `player.fields` as a cumulative total. These will all break or produce misleading output after 21.1. A full audit of every read of `player.fields` in engine, store, and components is needed before 21.1 is merged.

- _Agreed._

**3. Save file compatibility**
Existing saves have cumulative `player.fields` values and `TechState` entries without `fieldProgress`. The migration default (all zeroes) is safe but means loaded games restart tech progress from zero. This may be acceptable given the model change, but warrants a clear warning to the user (or an explicit save-wipe notice).

- _I am happy to break save game compatiability in this phase_

**4. The Rumour → Progress threshold numbers**
"All required fields ≥ 33% OR 33% of fields ≥ 50%" are placeholder numbers from the draft. These need to be defined as named constants and balanced against the distribution algorithm. The risk is techs advancing to Progress very quickly in early turns if field output is high relative to recipe thresholds.

- _The risk is recognised. Playtesting to assess the impact. Set named constants for these thresholds._

**5. Research setback events — which tech is affected?**
The draft examples ("fire in Research Lab #2 sets back physics for technology X") require events to reference a specific tech. Currently `EventDef` has no `targetTechId` field, and the event system has no mechanism to name a specific tech. This needs design work before setback events can be implemented. Suggested approach: setback events target the in-progress tech with the most accumulated `fieldProgress` in the relevant field (i.e., the one with the most to lose), and name it explicitly in the news item. This makes the event feel purposeful rather than arbitrary.

- _Agree with the suggested approach - the technology with the most progress in that field is the target of the event. This event cannot have a high likelihood. If two techs equally match, select the first returned._

**6. Signal decode decoupling**
The draft notes that Signal decoding will have its own progress/unlock mechanisms (TBC). The current signal system reads `player.fields.physics` and `player.fields.mathematics` to compute decode rate. After 21.1 these values become per-turn output rather than cumulative totals. The signal decode rate formula will need revisiting — using per-turn output as the input is probably the right model, but the numbers will need rebalancing.

- _Understood and I am happy to break Signal decoding at this stage; it should not be possible during Era 1 anyway._

**7. Tech recipes with 4–6 fields and existing recipe variance**
The current `recipeVariance` system applies ±N% to each field independently. With 5-field recipes, variance in one field can make that field's threshold much harder or easier without affecting the others, potentially creating unbalanced outlier recipes. Consider whether variance should be applied to the total recipe cost and then distributed, rather than per-field independently.

- _Interesting observation. I always imagined that, since we have an alt-history setting, that there should be some 'unexpected' science requirements for technologies, and that's why I wanted the variance. Let's stay with the current design for now._

**8. Breakthrough condition design**
Only 2–3 techs are breakthroughs. The specific threshold values (e.g., "20 Physics AND 3 Computing Hubs") need to be calibrated against typical mid-game facility builds. Too low and the breakthrough fires trivially early; too high and it never fires in practice. These will require significant playtesting.

- _Will definitely need testing. All the numbers are subject to change._
